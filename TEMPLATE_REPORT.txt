============================================================
3) APP ENTRYPOINT FLOW
============================================================
Runtime boot sequence:
1. index.html loads src/main.jsx.
2. main.jsx mounts the app and renders RouterProvider.
3. RouterProvider uses route definitions from src/Router/Router.jsx.
4. Router selects one of three layout shells (Main, Main2, Main3).
5. Each layout renders shared shell + Outlet content.

Important files:
- src/main.jsx
- src/Router/Router.jsx
- src/Main/Main.jsx
- src/Main/Main2.jsx
- src/Main/Main3.jsx

============================================================
6) PAGE COMPOSITION MODEL
============================================================
Home pages are section-composed (no CMS, no API):
- Home1 assembles sections from src/Componant1/*.
- Home2 assembles mostly src/Componant2/* plus Blog from Componant1.
- Home3 assembles mostly src/Componant3/* plus Brand from Componant2.

Inner pages follow a recurring pattern:
- Breadcamp hero at top.
- 1-3 content sections, often reused from Componant1/2 blocks.
- Example: AboutInner uses About + Support + Process + Team + Testimonial.

Design implication:
- The template is intentionally modular and section-first.
- You can swap, reorder, or remove sections quickly to create new page variants.

============================================================
8) DATA AND STATE ARCHITECTURE
============================================================
Current data model:
- No backend/API integration by default.
- No global state manager (Redux/Zustand/Context-based app state) in template core.
- Content mostly hardcoded inside component-level arrays/props.
- Many forms are UI-only placeholders.

What this means for customization:
- To make content editable/scalable, move static arrays into a data layer (JSON/CMS/API).
- Add validation and submission logic for contact/appointment/newsletter forms.
